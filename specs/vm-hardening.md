## 1. Executive Summary

The March 15, 2026 catch-up run from a Claude Cowork VM produced a
diagnostic report identifying **15 distinct failures** across the
catch-up workflow, plus **1 systemic silent failure** (token lifecycle)
identified during post-mortem. These cluster into five root causes
that are independent but interact:

| Root Cause | Issue Count | Severity | Class |
|------------|:-----------:|----------|-------|
| VM sandbox hard constraints (read-only FS, no `age`, no keyring) | 3 | P0 | Environmental |
| Configuration setup incompleteness (4-line user_profile, no health-check history) | 5 | P1 | Operational |
| Instruction file exceeds LLM consumption threshold (78KB, 1,422 lines) | 4 | P1–P2 | Architectural |
| No structural enforcement of workflow step execution | 3 | P1 | Design |
| Silent token expiry — connectors fail without user visibility | 1 | P1 | Operational |

### Architectural Diagnosis

The system's pain points break down across the Analysis Framework pillars:

| Pillar | Finding |
|--------|---------|
| **Bottleneck** | I/O-bound: the AI cannot read the full instruction file (78KB) in one pass. Not CPU/memory/DB. |
| **Data Consistency** | Eventual consistency is acceptable — state files update at end of catch-up, not transactionally. VM sessions produce zero state writes (by design in degraded mode). |
| **Component Coupling** | `Artha.md` is a **god-object** — identity, workflow (21 steps), routing table, privacy rules, formatting specs, and action schemas all in one file. The AI CLI systems read it as a single blob. Decomposition into bounded contexts (identity, workflow phases, reference tables) is the correct intervention. |

### Strategic Decision

This plan follows the **Strangler Fig Pattern**: each phase wraps one
existing behavior with a new layer, validates it independently, and only
then proceeds to the next. No "Big Bang" rewrite. The old monolithic
Artha.md remains functional behind a feature flag until all 5 phases
prove stable.

## 3. Root Cause Analysis — Issue Dependency Graph

The 15 issues are not independent. They cascade from 5 root nodes:

```
ROOT 1: user_profile.yaml is a 4-line skeleton
    │

    ├─► No integrations.microsoft_graph → connectors stay disabled (#5)
    ├─► No household config → briefing format defaults wrong
    │
    └─► health-check.md has no last_catch_up timestamp
            │
            └─► hours_elapsed = undefined → digest mode detection fails (#7, Step 2b)

ROOT 2: age not installed in VM
    │
    ├─► vault.py decrypt fails → 10 .age state files inaccessible (#2)
    │       └─► Immigration, Finance, Health, Insurance domains blind
    │               └─► Cross-domain reasoning data-starved (#14)
    │
    └─► vault.py encrypt fails → no state persistence → no audit trail (#3)

ROOT 3: state/ directory read-only in VM
    │
    ├─► No state writes → Steps 7, 7b, 8h, 16, 17 silently no-op (#3)
    ├─► No health-check update → next session sees same stale data
    └─► No audit.md → no accountability trail for session

ROOT 4: Artha.md too large (78KB, 1,422 lines)
    │
    ├─► AI read ~800 lines → Steps 9–21 never ingested (#15)
    ├─► §8 (briefing format) not loaded → format improvised (#15)
    ├─► §3 (routing table) not loaded → ad-hoc domain routing (#9)
    └─► §4 (privacy rules) not loaded → PII suppression skipped (#11)

ROOT 5: No structural workflow enforcement
    │
    ├─► Preflight never executed → P0 failures not surfaced (#1)
    ├─► State files not read → Steps 1–4b skipped (#7, #8)
    ├─► Emails processed from snippets → incorrect data extraction (#10)
    └─► Slack single-query zero results accepted without retry (#12)

ROOT 6: Token lifecycle has no proactive management
    │
    ├─► MS Graph access token expires hourly; preflight only READS expiry, never REFRESHES (#4, #16)
    │       └─► check_msgraph_token() says "expired 59m ago — run --reauth"
    │              └─► User expected to detect + act → Outlook stays offline silently
    │
    ├─► MS Graph refresh token: 90-day rolling window, no usage tracking
    │       └─► If unused 90 days → refresh token silently dies
    │              └─► User must full re-auth (interactive browser flow)
    │
    ├─► Google tokens: proactive refresh in preflight ✓ (via validate_token_freshness)
    │       └─► Asymmetry: Google self-heals, MS Graph doesn't
    │
    └─► No end-of-briefing failure summary → user unaware connectors are offline (#16)
            └─► Outlook/To Do can stay disconnected for weeks without notice
```

## 4. Design Proposals & Trade-Off Matrices

### 4.1 Environment Detection — 3 Options

| Criterion | Option A: Inline in preflight.py | Option B: Separate detect_environment.py | Option C: Config `runtime_mode: vm` |
|-----------|:------:|:------:|:------:|
| SRP compliance | ✗ Violates | ✓ Clean | ✗ Manual |
| Auto-detection accuracy | Medium | High (multi-signal) | N/A (manual) |
| Risk of weakening local gate | **High** (auto-downgrade P0s) | **Low** (explicit flag) | Low |
| False-positive VM detection | Medium | Low (defensive default) | Zero |
| Implementation cost | Low (2h) | Medium (4h) | Trivial (1h) |

**→ Decision: Option B.** Preflight's gate contract stays strict. Advisory
mode is explicitly requested, never auto-inferred. Rationale in TD-1.

### 4.2 Instruction File Strategy — 3 Options

| Criterion | Option A: Keep monolithic, reorder | Option B: Decompose into workflow/*.md | Option C: On-demand TOC extraction |
|-----------|:------:|:------:|:------:|
| Fixes "AI reads 800/1422 lines" | ✗ Reordering helps one command | ✓ Each file < 25KB | ✗ Unreliable self-navigation |
| Token efficiency | Poor (78KB always) | Good (15KB base + phase on demand) | Medium |
| Light commands skip workflow | ✗ | ✓ (§R router) | ✗ |
| Step drift risk | N/A | Medium | N/A |
| Backward compat | ✓ | ⚠️ Needs §R router + `--no-compact` rollback | ✗ |

**→ Decision: Option B.** Empirical evidence is conclusive: monolithic
fails. `--no-compact` flag provides immediate rollback. Rationale in TD-2.

### 4.3 Read-Only Mode Strategy — 3 Options

| Criterion | Option A: VMs unsupported | Option B: First-class degraded | Option C: tmp/ + sync later |
|-----------|:------:|:------:|:------:|
| User gets VM briefing | ✗ No | ✓ Yes (labeled) | ✓ Yes (full) |
| Complexity | Zero | Low (skip-list) | High (reconciliation) |
| Security risk | None | Low (PII scan limited) | Medium (unencrypted tmp/) |
| Maintenance | None | Low | High |

**→ Decision: Option B.** Cowork VM is a real environment. Degraded-but-honest
briefing beats no briefing. Rationale in TD-3.

### 4.4 Workflow Compliance Strategy — 3 Options

| Criterion | Option A: Prompt checklists | Option B: Python orchestrator | Option C: Post-hoc audit |
|-----------|:------:|:------:|:------:|
| Preserves "prompts = logic" | ✓ | ✗ Architecture change | ✓ |
| Compliance guarantee | Probabilistic (~70–85%) | Deterministic (100%) | Reactive |
| Fits Twelve-Factor | ✓ | ✗ | ✓ |
| Implementation effort | Low | High | Medium |

**→ Decision: Option A (primary) + Option C (supplementary).** Phase gates
in workflow files maximize compliance probability. Post-hoc audit (Phase 5)
catches what the gates miss. Python orchestrator rejected — it would change
what Artha fundamentally is. Rationale in TD-4.

### 4.5 Token Lifecycle Management — 3 Options

| Criterion | Option A: Proactive refresh in preflight (parity with Google) | Option B: Background launchd daemon | Option C: Manual `--reauth` (status quo) |
|-----------|:------:|:------:|:------:|
| Fix MS Graph hourly expiry | ✓ (refresh_token lasts 90 days) | ✓ | ✗ (user must detect + act) |
| User effort | Zero (if catch-up runs regularly) | Zero | High (detect + terminal + browser) |
| Implementation cost | Low (~20 lines) | High (daemon, plist, privilege) | Zero |
| Refresh token tracking (90-day) | Add `_last_refresh_success` | Same | ✗ No tracking |
| Over-engineering risk | Low | **High** (daemon for personal tool) | N/A |
| Works cross-platform | ✓ | ✗ macOS only (launchd) | ✓ |

**→ Decision: Option A.** `check_msgraph_token()` gains proactive refresh
identical to Google's `check_token_freshness()`. Access tokens auto-refresh
every run; users only need `--reauth` when the 90-day refresh token dies.
Refresh token age tracked via `_last_refresh_success` in token file.
Rationale in TD-7.

### 4.6 Silent Failure Surfacing — 3 Options

| Criterion | Option A: Footer summary block in briefing | Option B: Separate `/health` report | Option C: Push notification (email/Slack) |
|-----------|:------:|:------:|:------:|
| Visible without extra command | ✓ Always shown | ✗ Must run `/health` | ✓ But noisy |
| Zero-effort discovery | ✓ | ✗ | ✓ |
| Actionable remediation inline | ✓ | ✓ | Limited (no terminal) |
| Implementation cost | Low (prompt content) | Low (already exists) | Medium (channel setup) |
| Works in degraded mode | ✓ stdout-only | ✗ May need state | ✗ Needs network |

**→ Decision: Option A (primary) + Option B (supplementary).** Every briefing
ends with a **Connector & Token Health** block listing offline connectors,
expired tokens, and actionable fix commands. Users cannot miss it. `/health`
remains available for on-demand deep inspection. Rationale in TD-8.

### Phase 1: Environment Detection Layer (Week 1)

**Scope:** New `scripts/detect_environment.py` — probes runtime environment
and returns a JSON capability manifest. This phase wraps nothing — it adds a
new pre-Step-0 capability. The existing workflow continues identically.

#### 1.1 Module: `scripts/detect_environment.py`

**Bounded context:** Environment sensing. SRP: probe and report facts.
No decision-making — consumers (preflight, workflow) decide.

**Contract:**

```python
@dataclass
class EnvironmentManifest:
    """Immutable snapshot of runtime capabilities."""
    environment: str              # "local_mac" | "local_windows" | "local_linux" | "cowork_vm" | "unknown"
    capabilities: dict[str, bool] # filesystem_writable, age_installed, keyring_functional,
                                  # network_google, network_microsoft, network_apple
    degradations: list[str]       # Human-readable list: "vault_decrypt_unavailable", etc.
    detection_signals: dict       # Raw signal data for debugging

def detect() -> EnvironmentManifest:
    """Run all probes. Idempotent and side-effect-free."""

def detect_json() -> str:
    """CLI entry point — prints manifest as JSON to stdout."""
```

**Detection heuristics (multi-signal, defense-in-depth):**

| Signal | Method | Weight | False-Positive Mitigation |
|--------|--------|:------:|--------------------------|
| Cowork marker | `os.path.isdir("/var/cowork")` OR `$COWORK_SESSION_ID` | Primary | Not classified as Cowork unless marker present |
| Read-only FS | `open(state/.env_probe, 'w')`, catch `EPERM`/`EROFS` | Primary | Probes `state/` specifically |
| `age` installed | `shutil.which("age")` | Independent | Checked regardless of classification |
| Keyring backend | `keyring.get_password("artha-probe", "env")` | Independent | Linux headless detection |
| Network: Google | `socket.create_connection(("gmail.googleapis.com", 443), timeout=3)` | Supplementary | TCP SYN only, no credentials |
| Network: Microsoft | `socket.create_connection(("graph.microsoft.com", 443), timeout=3)` | Supplementary | Timeout = blocked |
| Network: Apple | `socket.create_connection(("imap.mail.me.com", 993), timeout=3)` | Supplementary | Combined with marker for confidence |

**Classification logic (defensive — ambiguity defaults to strict):**

```python
if cowork_marker_present:
    environment = "cowork_vm"
elif platform == "Darwin":
    environment = "local_mac"
elif platform == "Windows":
    environment = "local_windows"
elif platform == "Linux":
    if filesystem_writable and age_installed and keyring_functional:
        environment = "local_linux"
    elif not filesystem_writable:
        environment = "cowork_vm"  # Strong: read-only FS + Linux
    else:
        environment = "unknown"    # Defensive: default strict
else:
    environment = "unknown"
```

**Caching:** `tmp/.env_manifest.json`, 5-minute TTL. Cold probe cost: ~2s
worst case (network timeouts). Cached: <1ms.

### Phase 2: Preflight Hardening (Week 1–2)

**Scope:** Four additive changes to `scripts/preflight.py`, one new state
template, and one content addition to `Artha.core.md`. All changes are
opt-in or extend existing paths. Default behavior unchanged.

**Strangler:** `run_preflight()` runs identically when no new flags passed.
Advisory mode, profile check, and health-check seeding are new code paths
activated by `--advisory` flag or `--fix`.

#### 2.1 Advisory mode (`--advisory` flag)

Add `--advisory` to `argparse` in `main()`:

```python
parser.add_argument(
    "--advisory",
    action="store_true",
    help=(
        "Run all checks but report P0 failures as ADVISORY instead of "
        "NO-GO. Use in sandboxed environments where hard blocks are "
        "known and accepted. NEVER use on local machines."
    ),
)
```

**Behavioral change:** All P0 checks still execute. In advisory mode:
- P0 failures reclassified to `"ADVISORY"` severity in output
- Exit code always 0 (never blocks)
- JSON adds `"advisory_mode": true`, `"degradation_list": [...]`
- Console: `⚠️ ADVISORY:` prefix instead of `⛔ BLOCKED:`

**Implementation:** Advisory mode wraps the output formatter in `main()`,
NOT the check runners. `run_preflight()` returns the same `list[CheckResult]`.
Only `main()` changes how it interprets results and sets exit code.

**What does NOT change:**
- `run_preflight(auto_fix, quiet)` signature
- All check functions
- Default behavior (no `--advisory`): exit 1 on P0 failure
- `--fix`, `--json`, `--quiet`, `--first-run` modes

#### 2.4 Token lifecycle management (proactive refresh + tracking)

**Problem:** Google tokens auto-refresh during preflight via
`validate_token_freshness(proactive_refresh=True)`. MS Graph tokens do NOT —
`check_msgraph_token()` only reads the expiry field and tells the user to
run `--reauth`. This asymmetry means Outlook/To Do can silently go offline
for hours or weeks without the user ever realizing it.

**Three-layer fix:**

**Layer 1 — Proactive refresh in preflight (~20 lines):**

Modify `check_msgraph_token()` to call `ensure_valid_token()` before
reporting expiry:

```python
def check_msgraph_token() -> CheckResult:
    # ... existing file-exists and JSON checks ...

    # --- NEW: proactive refresh (Layer 1) ---
    if secs_left < TOKEN_EXPIRY_WARN_SECONDS:
        try:
            from setup_msgraph_oauth import ensure_valid_token
            refreshed_token = ensure_valid_token(warn_seconds=TOKEN_EXPIRY_WARN_SECONDS)
            # Re-parse expiry from refreshed token
            new_expiry = refreshed_token.get("expiry", "")
            if new_expiry:
                new_dt = _dt.fromisoformat(new_expiry.rstrip("Z"))
                if new_dt.tzinfo is None:
                    new_dt = new_dt.replace(tzinfo=_tz.utc)
                new_secs = (new_dt - _dt.now(_tz.utc)).total_seconds()
                if new_secs > 0:
                    return CheckResult(
                        "Microsoft Graph token", "P1", True,
                        f"Valid for {int(new_secs/60)}m ✓ (just refreshed)",
                    )
        except Exception:
            pass  # Fall through to expiry warning
    # --- END Layer 1 ---
```

The `ensure_valid_token()` function already exists in `setup_msgraph_oauth.py`
and handles MSAL silent refresh via `acquire_token_by_refresh_token()`. This
just calls it from preflight — same pattern as Google's proactive refresh.

**Behavioral change:** If the refresh token is alive (< 90 days), the access
token silently auto-refreshes every preflight run. Users only need `--reauth`
when the 90-day refresh token has expired — reduced from "every hour" to
"once every ~90 days" (or never, if catch-ups run regularly).

**Layer 2 — Refresh token age tracking (~15 lines in `ensure_valid_token`):**

Track the last time the refresh token was successfully used:

```python
# In ensure_valid_token(), after successful silent refresh:
new_token["_last_refresh_success"] = datetime.now(timezone.utc).isoformat()
_save_token(new_token, client_id=client_id)
```

In `check_msgraph_token()`, read `_last_refresh_success` and warn at
**60 days** (30 days before the 90-day cliff):

```python
last_refresh = token_data.get("_last_refresh_success", "")
if last_refresh:
    last_dt = _dt.fromisoformat(last_refresh.rstrip("Z"))
    days_ago = (_dt.now(_tz.utc) - last_dt).days
    if days_ago > 60:
        return CheckResult(
            "Microsoft Graph token", "P1", False,
            f"⚠️ Refresh token last used {days_ago}d ago — expires at 90d. "
            f"Run catch-up from Mac to keep it alive.",
        )
```

This surfaces the 90-day cliff **30 days early**, giving users time to
run a catch-up from Mac rather than discovering full re-auth is needed.

Abstract all providers behind a `TokenProvider` protocol. Deferred until
a 4th provider arrives. Current 3-provider landscape doesn't justify the
abstraction.

## 7. Trade-Off Decision Log

### TD-1: Separate detect_environment.py vs. inline in preflight.py

**Chosen: Separate module.** Preflight's contract is go/no-go gating.
Detection is a separate concern. Mixing them would either weaken the gate
(auto-downgrade P0s on local machines) or require complex branching.
**Trade-off accepted:** Extra module (+180 lines), ~100ms cold startup cost.
**Reversibility:** Delete one file, zero impact.

### TD-2: Decompose Artha.md vs. keep monolithic

**Chosen: Decompose.** Empirical evidence: AI read 800/1,422 lines on
March 15. 78KB exceeds reliable LLM consumption. Reordering doesn't solve
catch-up (all steps matter). **Trade-off accepted:** Breaks "single file =
application" philosophy (TS §1.1). Introduces "file not loaded" failure mode.
`--no-compact` provides immediate rollback.
**Why alternatives failed:** Option A (reorder) doesn't help catch-up where
all 20 steps are needed. Option C (AI self-navigates TOC) is unreliable.

### TD-3: First-class read-only mode vs. VMs unsupported

**Chosen: First-class degraded mode.** Cowork VM is a real environment.
Degraded-but-honest briefing (Gmail+GCal, clearly labeled) beats no briefing.
Skip-list is simple condition check, not parallel codepath.
**Trade-off accepted:** PII scan limited in VM mode. Second execution path
for write steps. Footer explicitly labels limitation.
**Why Option C (tmp/ + sync) rejected:** State reconciliation is O(n)
complexity for O(1) benefit. New security surface (unencrypted temp state).

### TD-4: Prompt checklists vs. Python orchestrator

**Chosen: Prompt-based gates (supplemented by post-hoc audit).** Artha is
prompt-driven — the instruction file IS the application. Python orchestrator
would fundamentally change the architecture: orchestrator becomes app,
prompts become data. This contradicts DDD (workflow logic belongs in prompt
layer) and Twelve-Factor (config in config/).
**Trade-off accepted:** Compliance is probabilistic (~70–85%), not guaranteed.
Mitigated by 5 independent gates + Phase 5 post-hoc audit.

### TD-5: Mandatory email body reads vs. snippet tolerance

**Chosen: Mandatory for top 10.** For immigration/finance/health, snippet-
derived data (names, amounts, dates) may be wrong. False data in a briefing
is worse than no data. **Trade-off accepted:** ~5–15s added latency.
Correctness > speed for high-stakes domains.

### TD-6: MCP data PII treatment in VM mode

**Chosen: Note limitation in footer; accept for degraded mode.** Pipeline.py
can't write in read-only VM. MCP returns data in-context (works). In-context
piping through pii_guard.py adds complexity and fragility. Deferred to
future iteration. **Trade-off accepted:** VM briefings have weaker PII
filtering. Footer labels it. Briefing is stdout-only, not persisted.

### TD-7: Proactive MS Graph refresh vs. manual `--reauth` vs. daemon

**Chosen: Proactive refresh in preflight (parity with Google).** The
`ensure_valid_token()` function already exists and works — it just wasn't
called from preflight. Google tokens get `validate_token_freshness(
proactive_refresh=True)` during preflight; MS Graph tokens didn't. This is
a 20-line parity fix, not a new mechanism.
**Trade-off accepted:** Preflight gains an HTTP call to Microsoft's token
endpoint (~200ms). Acceptable — Google already does this twice (Gmail +
GCal). Total preflight time increases from ~2s to ~2.2s.
**Why daemon rejected:** Over-engineered for a personal tool that runs
catch-up at least weekly. Catch-up IS the heartbeat that keeps the refresh
token alive. A daemon solves a problem that the parity fix already solves.
**Why manual `--reauth` rejected:** Expecting users to detect token expiry
is hostile UX. The March 15 diagnostic proved this — MS Graph was offline
and the user didn't know until the post-mortem. Silent failures are
dangerous in a system whose value proposition is awareness.

### TD-8: Silent Failure Summary Block in every briefing

**Chosen: Mandatory footer table in every briefing.** The core insight:
if a connector fails silently, the user doesn't know what they're NOT
seeing. A briefing that looks complete but is missing Outlook mail creates
a false sense of coverage. The block makes this visible.
**Trade-off accepted:** ~5–10 lines added to every briefing, even when all
connectors are healthy. This is intentional — the "all green" state builds
trust. Users learn to glance at the block and move on when healthy, but
immediately notice when something turns red.
**Why separate `/health` insufficient:** Users run `/catch-up`, not
`/health`. If the failure surface is in a command they don't run, it might
as well not exist. The block goes where the user already is.

## 13. Security Analysis

### Threat Model — Changes Introduced

| Threat | Phase | OWASP | Mitigation |
|--------|:-----:|-------|------------|
| Advisory mode used locally → P0s bypassed | 2 | Broken Access Control | Flag is never auto-inferred. detect_environment recommends; user/AI must explicitly pass `--advisory`. Misclassification to `unknown` defaults to strict. |
| Network probes leak environment info | 1 | SSRF | Socket connect only (Layer 4). No HTTP, no data sent, no credentials. DNS to well-known public domains. |

## Appendix C: Token Lifecycle — AS-IS vs. TO-BE

### AS-IS (before this plan)

| Provider | Access Token TTL | Refresh Token TTL | Auto-refresh at preflight? | Auto-refresh at runtime? | Silent failure visible? |
|----------|:----------------:|:-----------------:|:--------------------------:|:------------------------:|:-----------------------:|
| Google Gmail | ~1 hour | Indefinite | ✅ Yes (`validate_token_freshness`) | ✅ Yes (`Credentials.refresh`) | N/A (auto-heals) |
| Google Calendar | ~1 hour | Indefinite | ✅ Yes (same) | ✅ Yes (same) | N/A (auto-heals) |
| MS Graph | ~1 hour | 90 days rolling | ❌ **Read-only check** | ✅ Yes (`ensure_valid_token`) | ❌ **No** |
| iCloud | Static app password | Never expires | N/A | N/A | ❌ **No** |

**Gap:** MS Graph has the infrastructure for proactive refresh
(`ensure_valid_token()` exists and works) but preflight doesn't call it.
The function is only called at runtime by `pipeline.py` and `todo_sync.py`,
which may not run if preflight blocks first.

### TO-BE (after Phase 2 + Phase 4)

| Provider | Access Token TTL | Refresh Token TTL | Auto-refresh at preflight? | Refresh token age tracked? | Silent failure visible? |
|----------|:----------------:|:-----------------:|:--------------------------:|:--------------------------:|:-----------------------:|
| Google Gmail | ~1 hour | Indefinite | ✅ Yes (unchanged) | N/A (no expiry) | ✅ Briefing footer |
| Google Calendar | ~1 hour | Indefinite | ✅ Yes (unchanged) | N/A (no expiry) | ✅ Briefing footer |
| MS Graph | ~1 hour | 90 days rolling | ✅ **Yes (parity fix)** | ✅ **60-day warning** | ✅ **Briefing footer** |
| iCloud | Static app password | Never expires | N/A | N/A | ✅ **Briefing footer** |

**Result:** All providers self-heal or surface failures. No silent offline state.
Users need `--reauth` only for: (1) MS Graph refresh token > 90 days stale,
(2) OAuth consent revoked, (3) first-time setup.
